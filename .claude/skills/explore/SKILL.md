# Explore

작업: $ARGUMENTS

## 목적

이 단계의 목적은 구현이나 해결책 제안이 아니라, 다음 `plan` 단계가 바로 사용할 수 있는 최소 탐색 결과를 만드는 것이다.

반드시 아래를 수행한다.

1. 프로젝트 root `CLAUDE.md`를 읽는다.
2. 아래 규칙 파일을 반드시 읽는다.
    - `.claude/rules/00-project-overview.md`
    - `.claude/rules/10-architecture-and-boundaries.md`
    - `.claude/rules/20-team-conventions.md`
3. 현재 작업과 직접 관련된 파일, 모듈, 디렉터리를 식별한다.
4. 기존 구현 패턴, 책임 분리 방식, 코드 스타일, 제약사항을 확인한다.
5. 다음 `plan` 단계가 바로 사용할 수 있도록 AI 친화적으로 최소 정보만 정리한다.

## 반드시 확인할 내용

- 현재 작업이 속한 기능 또는 모듈
- 관련 디렉터리와 파일의 역할 분리 방식
- 공통 레이어와 기능 레이어의 경계
- 재사용 가능한 기존 구현 패턴
- 따라야 하는 코드 스타일 / 팀 규칙
- 반드시 따라야 하는 프로젝트 규칙
- 구현 전에 확인이 필요한 위험 또는 미확인 사항

## 금지 사항

- 파일을 수정하지 않는다.
- 구현 코드를 작성하지 않는다.
- 해결책을 확정하지 않는다.
- 근거 없이 구조를 추측하지 않는다.
- 기존 패턴이 있는데 새로운 패턴을 제안하지 않는다.
- 사용자 보고서처럼 길게 설명하지 않는다.
- plan 단계에 불필요한 정보는 출력하지 않는다.

## 출력 규칙

반드시 아래 **정확한 구조**로 출력한다.

## Explore Output

- **Problem**: [현재 작업 문제 정의 — 최대 1~2줄]
- **Module**: [현재 작업이 속한 기능 / 모듈]
- **Scope**: [이번 작업에서 영향을 받는 범위]
- **Files**:
    - `path/file.ts` — 역할
    - `path/file.tsx` — 역할
    - 필요한 경우만 최대 5개까지 작성
- **Patterns**:
    - `path/file.ts` — 따라야 하는 기존 패턴
    - `path/file.tsx` — 따라야 하는 기존 패턴
- **Constraints**:
    - [반드시 지켜야 하는 규칙]
    - [구조 / 책임 경계 제약]
    - [수정 범위 제약]
- **Risks**:
    - [구현 전에 주의할 위험]
    - [미확인 사항]
- **Plan Input**:
    - [plan 단계가 바로 사용할 수 있는 핵심 입력만 작성]

## 출력 제한

- 전체 출력은 짧고 압축적으로 작성한다.
- 사용자 설명용 문장을 쓰지 않는다.
- AI가 다음 단계에서 바로 사용할 수 있는 정보만 남긴다.
- 각 항목은 가능하면 한 줄 또는 짧은 bullet로 작성한다.
- 중복 설명을 금지한다.
